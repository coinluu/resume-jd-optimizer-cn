# Demo：数据分析师

> 场景：数据运营投递数据分析师。重点展示如何识别已有分析证据，同时保留 Python 和 A/B 测试缺口。

## 用户背景

- 当前岗位：某电商公司的数据运营。
- 工作时间：2024.01 至今。
- 目标岗位：数据分析师。
- 已知边界：使用 SQL 和 Excel；未使用 Python，也没有参与 A/B 测试。

## 原始简历片段

```text
某电商公司｜数据运营｜2024.01-至今

- 负责日报周报制作。
- 分析用户数据，为运营提供建议。
- 使用 SQL 和 Excel 取数。
```

## 目标 JD 片段

```text
- 建设业务指标体系和 BI 看板，支持经营决策。
- 使用 SQL/Python 完成用户行为、漏斗和专题分析。
- 参与 A/B 测试设计、评估和复盘。
- 能从业务问题出发形成分析建议并推动落地。
```

## Skill 诊断结果

| JD 要求 | 原始简历证据 | 状态 | 诊断 |
|---|---|---|---|
| SQL | “使用 SQL 和 Excel 取数” | 弱覆盖 | 证明使用过 SQL，但没有分析场景和深度 |
| 漏斗分析 | “分析用户数据” | 可补充 | 可能存在分析经历，需要追问 |
| 指标体系 / 看板 | “负责日报周报制作” | 弱覆盖 | 缺少指标数量、使用者和业务用途 |
| 推动建议落地 | “为运营提供建议” | 弱覆盖 | 缺少建议内容、是否采用和后续变化 |
| Python | 无 | 未覆盖 | 真实缺口，不应补写 |
| A/B 测试 | 无 | 未覆盖 | 真实缺口，不应补写 |

## 素材追问

1. 日报周报包含哪些核心指标？服务多少使用者？
2. 你是否定义或统一过指标口径？
3. 请提供一次使用 SQL 发现业务问题的完整过程。
4. 你提出了什么建议？运营是否采用？
5. 建议实施后的变化如何统计，周期和来源是什么？
6. 是否使用 Python、BI 工具或参与 A/B 测试？
7. 如何判断结果变化与建议相关，而不是其他因素导致？

## 用户确认的真实素材

- 为运营团队搭建 6 个核心漏斗指标的周报，服务 12 名运营同事。
- 使用 SQL 分析发现新用户在优惠券领取后的使用环节流失明显。
- 建议调整提醒节点。
- 运营实施后两周内，优惠券使用率从 21% 提升至 26%。
- 未使用 Python，也没有参与 A/B 测试。

## 面试追问

- 6 个漏斗指标分别是什么？口径如何确定？
- SQL 查询如何定位优惠券使用环节流失？
- 提醒节点为什么被认为是主要问题？
- 使用率从 21% 提升至 26% 的统计周期、用户范围和数据来源是什么？
- 同期是否有其他运营动作？如何解释归因边界？
- 为什么没有使用 Python？当前准备如何补足？
- 如果重新设计验证方案，你会如何设置对照或实验？

## 风险提示

- 使用率变化发生在建议实施之后，但不能直接证明严格因果关系。
- 没有 Python 和 A/B 测试证据，必须作为该 JD 的真实缺口。
- “搭建 6 个核心漏斗指标的周报”需要能解释指标定义和使用方式。
- 如果只是使用现有报表而非定义口径，应降低“搭建”表述强度。

## Demo 结论

本 Demo 将模糊的“制作周报、分析数据”重构为可追问的业务分析过程，同时保留了 Python、A/B 测试和因果归因方面的真实短板。
