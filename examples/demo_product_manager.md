# Demo：B 端产品经理

> 场景：产品专员投递 B 端产品经理。重点展示如何从职责化表达中挖掘真实项目证据。

## 用户背景

- 当前岗位：某企业服务公司的产品专员。
- 工作时间：2023.03 至今。
- 目标岗位：B 端产品经理。
- 已知边界：会使用 SQL 做基础查询，但不应写成“熟练 SQL”。

## 原始简历片段

```text
某企业服务公司｜产品专员｜2023.03-至今

- 负责订单系统需求收集和产品迭代。
- 对接客户和研发，推动需求上线。
- 参与数据分析和产品优化。

项目：订单审批流程改版
- 调研客户需求，输出原型和 PRD。
- 跟进研发测试和上线。
```

## 目标 JD 片段

```text
- 负责企业客户订单管理产品的需求分析、方案设计和迭代。
- 能独立输出 PRD 和原型。
- 有复杂流程或订单类产品经验。
- 具备数据分析与跨部门推进能力。
- 熟悉 SQL 者优先。
```

## Skill 诊断结果

| JD 要求 | 原始简历证据 | 状态 | 诊断 |
|---|---|---|---|
| B 端订单产品经验 | “订单系统需求收集和产品迭代” | 弱覆盖 | 有场景，但没有说明复杂度和本人负责范围 |
| PRD 和原型 | “输出原型和 PRD” | 弱覆盖 | 缺少是否独立负责及交付结果 |
| 跨部门推进 | “对接客户和研发” | 弱覆盖 | 缺少协作对象、阻碍和推进结果 |
| 数据分析 | “参与数据分析” | 弱覆盖 | 没有分析过程和结论证据 |
| SQL | 技能区出现 SQL | 弱覆盖 | 只能证明用户声明使用过，不能证明熟练程度 |

主要问题：

- 句子像岗位职责，不像成果证据。
- “负责、推动、参与”没有对应的负责范围。
- 没有写出订单审批流程的复杂度。
- 没有结果口径，无法判断上线产生了什么变化。

## 素材追问

为了避免编造，Skill 应先追问：

1. 订单审批流程服务什么类型的企业客户？主要问题是什么？
2. 你访谈了多少客户，如何归纳和取舍需求？
3. PRD、原型、验收和上线跟进中，哪些由你本人负责？
4. 你协同了哪些角色，各自如何分工？
5. 改版前后流程是否有可验证变化？
6. 上线后是否有客服工单、使用反馈或其他结果？
7. SQL 具体用于什么任务，使用深度如何？

## 用户确认的真实素材

- 独立访谈 8 家企业客户，发现多级审批配置复杂是主要问题。
- 负责 PRD、原型和验收。
- 协同 5 名研发及 1 名设计。
- 配置步骤从 9 步减少到 5 步。
- 上线后一个月，相关客服工单较改版前下降约 30%。
- SQL 仅用于基础业务查询。

## Before / After

| Before | After |
|---|---|
| 负责订单系统需求收集和产品迭代。 | 围绕企业客户订单审批场景，独立访谈 8 家客户并梳理多级审批配置问题，负责需求分析、PRD、原型与验收。 |

## 面试追问

- 8 家客户如何选择？需求冲突时如何取舍？
- 多级审批配置的主要问题是什么？为什么选择该改版方案？
- 你与研发、设计分别如何分工？你解决过什么推进阻碍？
- 工单下降约 30% 的统计周期和口径是什么？
- 如何排除其他因素对工单下降的影响？
- 基础 SQL 查询具体包含哪些操作？为什么不能称为熟练 SQL？

## 风险提示

- 工单下降约 30% 需要能够解释数据来源、统计周期和归因边界。
- 用户历史岗位名称仍应保留“产品专员”，不能改成“B 端产品经理”。
- SQL 只用于基础查询，不应写成“熟练 SQL”或独立数据分析能力。
- “推动上线”不等于主导整个订单系统；必须保持项目范围边界。

## Demo 结论

本 Demo 的优化来自用户确认的客户访谈、负责范围、协作规模和结果数据。Skill 没有创造新项目，也没有把基础 SQL 包装成熟练能力。
