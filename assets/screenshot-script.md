# Demo 截图脚本

本文件告诉维护者应该截哪些图、每张图展示什么，以及如何避免隐私和夸大风险。

## 截图准备

### 使用材料

- 使用 [产品经理 Demo](../examples/demo_product_manager.md)、[数据分析师 Demo](../examples/demo_data_analyst.md) 和 [转行 AI 产品经理 Demo](../examples/demo_ai_product_transition.md)。
- 不使用真实求职者简历。
- 不出现真实电话、邮箱、公司、客户或未公开数据。
- 截图前关闭浏览器书签、账号头像、侧边栏私人会话等无关信息。

### 统一风格

- 建议宽度：1440 px 或 1600 px。
- 使用浅色背景和清晰等宽代码块。
- 每张图只展示一个主题。
- 保留项目名或 GitHub 仓库标题，避免大面积品牌水印。
- 不添加“包拿 Offer”“保证过 ATS”等宣传语。

## 截图 4：Before / After

### 目标

展示“先追问真实事实，再优化表达”。

### 使用材料

优先使用 [Before / After 展示稿](demo-before-after.md) 中的产品经理卡片。

### 必须展示

- 原始简历片段。
- Skill 素材追问。
- 用户确认后的优化结果。
- 一条风险标签。

### 避免

- 只展示 After，造成“一键润色”误解。
- 使用过长段落，导致移动端无法阅读。
- 使用无法解释的数据。

### 建议文件名

```text
assets/screenshots/before-after.png
```

## 截图 6：ATS 版本输出

### 目标

## 截图后检查

- [ ] 没有个人身份信息。
- [ ] 没有真实公司、客户或商业机密。
- [ ] 没有夸大结果或招聘效果承诺。
- [ ] 截图文字在 GitHub 页面和移动端可读。
- [ ] Before、追问、After 和风险之间逻辑一致。
- [ ] 图片文件名与 README 或文档引用一致。

## 推荐发布顺序

1. README 顶部。
2. Before / After。
3. JD 解析结果。
4. 简历缺口分析。
5. ATS 版本输出。
6. 面试追问清单。

这个顺序能先说明价值，再证明工作流和真实性边界。
