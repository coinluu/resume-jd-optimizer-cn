# Demo Before / After 展示稿

本文件用于 GitHub README、社交媒体、项目介绍和截图制作。所有内容均为脱敏示例，不代表招聘结果。

## 展示原则

- Before 保留真实但模糊的原始表达。
- After 只使用用户已确认事实。
- 同时展示 Skill 追问和风险提示，避免看起来像“一键润色”。
- 不使用大厂名称、夸张职级或无法解释的数据。

## 卡片 1：产品经理

### Before

```text
负责订单系统需求收集和产品迭代。
对接客户和研发，推动需求上线。
```

### Skill 追问

```text
- 订单流程的具体问题是什么？
- 你本人负责哪些交付物？
- 协同哪些角色？
- 上线后有什么可验证变化？
```

### After

```text
围绕企业客户订单审批场景，独立访谈 8 家客户并梳理多级审批配置问题，
负责需求分析、PRD、原型与验收。

协同 5 名研发及 1 名设计推动流程改版上线，
将配置步骤从 9 步减少至 5 步。
```

### 风险标签

```text
客服工单下降约 30% 需要解释统计周期和归因边界。
基础 SQL 不得写成熟练 SQL。
```

## 卡片 2：数据分析师

### Before

```text
负责日报周报制作。
分析用户数据，为运营提供建议。
```

### Skill 追问

```text
- 周报包含哪些指标，服务多少使用者？
- 使用 SQL 发现了什么业务问题？
- 建议是否被采用，后续如何变化？
- 是否参与 Python 或 A/B 测试？
```

### After

```text
面向 12 名运营同事搭建包含 6 个核心漏斗指标的周报，
统一日常跟踪口径并支持运营复盘。

使用 SQL 定位优惠券使用环节流失问题并建议调整提醒节点；
实施后两周内使用率从 21% 提升至 26%。
```

### 风险标签

```text
使用率变化不等于严格因果。
没有 Python 和 A/B 测试证据，不写入简历。
```

## 卡片 3：转行 AI 产品经理

### Before

```text
整理客户常见问题并支持交付团队。
推动课程平台功能优化。
```

### Skill 追问

```text
- 覆盖多少客户和问题？
- 是否建立过搜索结果检查规则？
- 哪些是可迁移经验，哪些 AI 经验确实缺失？
```

### After

```text
访谈 15 家企业客户并整理 120 条高频咨询，
按使用场景分类需求，支持帮助中心搜索优化方案设计。

建立每周 50 条结果的准确性抽查表；
三个月后客户重复咨询量较之前下降约 18%。
```

### 风险标签

```text
传统搜索优化不能写成 RAG。
无 AI 商业落地、RAG 或 Agent 项目经验。
```
