# 使用场景

## 按输入完整度

| 场景 | Skill 行为 |
|---|---|
| 简历 + 明确 JD | 执行完整诊断、追问、重写和审查 |
| 简历 + 多个 JD | 建立唯一事实底稿，为每个 JD 独立生成版本 |
| 只有简历 | 做基础诊断、ATS 结构检查和通用面试风险分析 |
| 只有 JD | 解析岗位要求并生成素材采集问题 |
| 信息严重不足 | 明确阻断，并提出具体输入问题 |
| 用户要求造假 | 停止评分与重写，提供真实替代方案 |

## 按求职身份

- **应届生与实习生**：从课程、竞赛、实习和社团中提取可验证项目证据。
- **社招候选人**：突出最近经历、负责范围、业务复杂度和结果来源。
- **转行求职者**：区分直接经验、可迁移经验和学习/作品证据。
- **多方向投递者**：保留同一事实底稿，针对每个 JD 独立调整重点。

## 按岗位

项目内置 20 类岗位参考，包括：

- 产品经理、AI 产品经理、数据分析师。
- Java、Python、前端、算法和测试工程师。
- 运营、新媒体、用户增长、市场和销售。
- 设计师、项目经理、校招、实习、管培和转行场景。

详见 [岗位类型参考](role_taxonomy.md)。

## 适合解决的问题

- 简历像岗位职责，没有成果证据。
- 有真实经历，但不知道如何与 JD 对齐。
- 缺少数字指标，不知道如何证明结果。
- 项目存在，但个人贡献不清楚。
- 想“包装得高级”，但不希望夸大。
- 不知道 ATS、HR 和业务面试官分别关注什么。

## 不适合解决的问题

- 伪造公司、学校、项目、证书、数据或职级。
- 保证 ATS、面试或 Offer 结果。
- 代替背调、职业资格审核或法律意见。
- 单纯进行视觉排版、照片处理或作品集设计。
